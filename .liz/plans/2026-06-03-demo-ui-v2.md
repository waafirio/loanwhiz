# Demo UI v2 (Next.js)

## Context & intent

### The problem

The v1 demo UI (epic #75, a unified Gradio app, now merged to main) proved the whole stack works end to end — but it has an aesthetic ceiling and live defects. The operator, running it, hit: an ugly default Gradio look, a `gr.BarPlot`-fed-a-list crash on Pool & Performance (`'list' object has no attribute 'to_json'`), empty fields (the deal doesn't auto-load; tabs are button-gated), and the cold-extraction "deal model not extracted" state. The operator's verdict: **"the whole UI needs to be re-built with a standard B2B SaaS design system — professional, clean lines, legible fonts and light theme."**

### Why a Next.js rebuild (and why now)

Gradio cannot deliver a real B2B SaaS look — themed, it gets cleaner/lighter but still reads as "a Gradio app," which the SF-practitioner judges (esp. Samuel Griek, who builds TrancheFlow commercially) would clock. The path that actually delivers it is a real frontend: **Next.js + Tailwind + shadcn/ui**, calling the **FastAPI REST API (#22) that already exists and is tested**.

Crucially, the calculus that originally favoured Gradio has flipped. Gradio was chosen for speed/low-risk when the backend wasn't built — standing up a separate React frontend *plus* backend *plus* their integration was a big new surface. That's gone: the backend is 100% done and the API already exposes `/query`, `/deal/{id}/model`, `/deal/{id}/compliance`, `/deal/{id}/project`. So v2 is **frontend-only against a finished, tested API** — a contained effort, and it matches Waafir's own Next.js stack.

### The hard constraint: LEAN, not over-engineered

The operator was explicit: **"I don't want it over-engineered — I just need a clean UI that doesn't annoy me so that I can focus on the backend."** This is a binding design constraint, not a nice-to-have. The whole point of the rebuild is a frictionless window into the backend so the operator can get back to backend work. So:

- **Lean entirely on shadcn/ui defaults** — we are NOT building a design system, we are *using* one that is already clean/professional/light out of the box. Using good defaults is the opposite of over-engineering.
- **Explicit non-goals (do NOT build):** no custom theming work (shadcn defaults as-is); no state-management library (Redux/Zustand/etc.) — plain `fetch` + React state; no auth; no routing cleverness; no animations; simple tables + basic Recharts, not bespoke visualizations.
- **Build it mostly autonomously** so the operator isn't pulled off backend work — surface only what genuinely needs them; hand back "done, here's how to run it."
- **Keep the orchestration itself lean** — a 4-child epic, not the 9-child design-system epic first sketched (which was itself the over-engineering the operator warned against).

### Architecture

```
Next.js 14 (App Router) + Tailwind + shadcn/ui   [light theme, shadcn defaults]
  → typed fetch client → FastAPI (#22) over HTTP (CORS enabled)
  → sidebar nav + top bar
  → 5 pages mirroring the backend: Overview · Pool · Waterfall · Compliance · Projection
  → chat: slide-over panel, streams from /query with citations
Two processes at demo time: uvicorn (API) + next (frontend).
```

### Alternatives considered and rejected

- **Heavily-themed Gradio** — fast (~1 day), reuses v1, but hard ceiling: still looks like Gradio, fails the "B2B SaaS" bar. Rejected.
- **Streamlit** — a third framework, nicer defaults than Gradio but still not product-grade; rejected.
- **Full Next.js design-system epic (9 children, custom theme tokens, etc.)** — exactly the over-engineering the operator vetoed. Rejected in favour of the lean 4-child shape using shadcn defaults.
- **Routing the frontend through new endpoints / a BFF layer** — unnecessary; the existing FastAPI is the contract, with at most a small CORS/endpoint tweak.

### Relationship to v1 (Gradio)

The v1 Gradio app stays on `main` as a working fallback (epic #75, PR #95 already merged). v2's final child retires it once v2 is demo-ready. No further investment in the v1 Gradio P0 bugs — they are throwaway.

### Demo target (10 June)

Operator runs two commands (uvicorn + next, or one wrapper script), opens the Next.js app, and walks the same 5-view narrative + chat — now clean, light, legible, and not annoying.

## Decomposition

### Epic: Demo UI v2 — Next.js frontend on the FastAPI   (umbrella #TBD)

A lean Next.js + Tailwind + shadcn/ui app that replaces the Gradio demo UI with a clean, light, professional surface over the existing FastAPI REST API. Built to a strict no-over-engineering constraint: shadcn defaults, plain fetch, no state libs, simple tables/charts.

- **App scaffold + design baseline + API client + shell** — Next.js 14 (App Router) + Tailwind + shadcn/ui (defaults, light theme), a typed `fetch` API client for the FastAPI endpoints, and the sidebar-nav + top-bar layout shell that pages render into. Defines the contract pages build against. Sequencing: parallel. Paths: `web/**` (app shell, lib/api client, layout, globals).
- **CORS on FastAPI + one-command run + README** — add CORS middleware to the FastAPI app so the browser frontend can call it; add a run script/Make target launching uvicorn + next together; brief README for running v2. Sequencing: parallel. Paths: `src/loanwhiz/api/main.py`, `web/README.md`, run script.
- **Five backend pages (Overview/Pool/Waterfall/Compliance/Projection)** — the five views, all the same pattern (fetch endpoint → shadcn table + basic Recharts chart): Overview (deal model card), Pool (3-period analytics), Waterfall (cascade + tranche distributions), Compliance (report verification + covenant grid + actual-vs-stress-vs-limit chart), Projection (base/stress 12-month + WAL). Loading + empty states so latency never reads as broken. Sequencing: sequential. After #<scaffold>. Paths: `web/app/**`, `web/components/**`.
- **Chat slide-over + retire Gradio** — an omnipresent slide-over chat panel streaming from `/query` with inline citations; then retire the v1 Gradio app (`clients/demo`, `demo/`) and point docs at v2. Sequencing: sequential. After #<scaffold>, after #<pages>. Paths: `web/components/chat/**`, `clients/demo/**`, `README.md`.
