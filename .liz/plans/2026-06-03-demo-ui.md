# Demo UI

## Context & intent

### The problem

LoanWhiz's six functional epics are all on `main` — the framework works end to end (validated: 570 tests green, a live 24-second `--fast` demo run). But the demo-facing surface is three **disconnected** Gradio apps (chat:7860, dashboard:7861, compliance:7862), each built standalone by the fleet, each calling primitives directly, with no shared narrative. For the 10 June demo day in front of Samuel Griek (Eagleyes.ai / TrancheFlow — builds this commercially), Mark Mearing-Smith (Jefferies CLO desk), Luca Borella (FINOS / Algoritmica, the challenge author), and a VC judge, three separate tool tabs tell no story. We need one cohesive demo UI that walks the deal-model differentiator front-to-back.

### Why this shape

The UI is driven by the **demo-day narrative**, not by "wrap the existing apps nicer":

> *"Here's a deal as a stack of PDFs and CSVs. Watch LoanWhiz turn it into a living model."*
> 1. Load Green Lion → ingest prospectus, extract waterfall + covenants automatically
> 2. Run the actual loan tapes through that extracted waterfall
> 3. Verify the computation against the real investor reports (did the servicer pay correctly?)
> 4. Monitor covenants + project forward under stress vs the trigger limits
> 5. Ask anything in natural language (chat, the free-exploration capstone)

That arc reuses everything the fleet built and ends on the deal-model capability no other team is likely to have.

### Decisions made with the operator (2026-06-03)

- **Stack: unify in Gradio.** Not a Next.js/React rebuild (too much new surface under time pressure) and not Streamlit (a third framework). Builds on the 3 Gradio apps the fleet already wrote. All-Python, low-risk, fastest path to a working guided demo.
- **Structure: one unified multi-tab app, chat accessible from every tab.** Tabs ordered to tell the story left-to-right but freely navigable; a chat panel is docked persistently so Q&A is reachable anywhere.
- **The unified app SUPERSEDES the 3 standalones** — chat/dashboard/compliance get absorbed as tabs and the standalone `clients/{chat,dashboard,compliance}/` apps are deleted to avoid confusion about which is "the" demo.
- **Fully live** — the app calls primitives/agent directly in-process (no dependency on the separate REST API #22 running, keeping demo-day failure surface to one process). Everything runs in real time…
- **…EXCEPT prospectus extraction, which is pre-warmed.** Hard constraint discovered during #73: a *cold* `extract_deal_model()` takes **>10 minutes** on this hardware because Docling's RapidOCR is CPU-only. So the Deal Overview tab reads the cached `deal_model.json` (instant) while everything else (tapes, waterfall, covenants, projection, chat) genuinely runs live. Pre-warming the extraction cache is a demo-day prerequisite captured in the runbook.
- **Demo-day guidance: a written runbook is sufficient** — no in-app scripted "guided mode" toggle.

### Known issue to fix as part of this epic

`ReportVerifier` (merged primitive, epic #12) instantiates a bare `genai.Client()` that defaults to the Gemini Developer API and needs `GEMINI_API_KEY` — it fails under Vertex/ADC unless `GOOGLE_GENAI_USE_VERTEXAI=true` is set. The #73 demo script worked around it with an env shim, but the Compliance tab will hit the same wall. Fix `ReportVerifier` to use the Vertex client like every other primitive, so the compliance tab works without env gymnastics.

### Deferred feature now folded in

The covenant/compliance **time-series charts** (previously parked in memory `loanwhiz-covenant-timeseries-idea`): plot the 3 actual monthly points + `CashflowProjector` base/stress forward paths against each trigger line. The actuals alone are flat/sparse (brand-new synthetic deal); the projector stress path approaching/breaching the limit is the compelling line, and it showcases the deal-model differentiator. This lives in the Compliance & Covenants tab.

### Architecture

```
clients/demo/app.py — single Gradio Blocks app
  ├── persistent Chat panel (docked, every tab) → planner agent (#20)
  └── Tabs (story order, freely navigable):
      1. Deal Overview      → cache-first deal_model (tranches/waterfall/triggers/completeness)
      2. Pool & Performance → 3-period tape analytics, EPC/geo/arrears
      3. Waterfall          → CollectionsAggregator → WaterfallRunner, cascade viz
      4. Compliance         → ReportVerifier + CovenantMonitor + time-series charts
      5. Cashflow Projection→ base/stress 12-mo, WAL
  shared "loaded deal" state object read by all tabs
  calls primitives/agent in-process (REST API #22 not required to be running)
```

### Alternatives considered and rejected

- **Next.js/React polished web app** — slickest, aligns with Waafir's stack, but a brand-new surface to build and debug in <7 days; rejected for risk.
- **Streamlit** — nicer default look than Gradio, but a third UI framework to introduce; rejected for marginal benefit over reusing the Gradio work.
- **Keep 3 separate apps, just polish** — lowest effort but disjointed on stage; rejected.
- **Route the UI through the REST API (#22)** — architecturally cleaner (exercises the service layer) but adds a second running process as a demo-day failure point; rejected in favour of in-process calls. The API keeps its own tests as proof of the service layer.
- **Fully-live cold extraction on stage** — rejected: >10 min Docling/RapidOCR hang. Pre-warmed cache instead.
- **In-app guided-mode toggle** — rejected as scope; a written runbook suffices.

### Demo target (10 June)

Operator pre-warms the extraction cache, launches `clients/demo/app.py`, and walks the 5-tab narrative live (extraction cached, everything else real-time), taking ad-hoc questions through the docked chat.

## Decomposition

### Epic: Demo UI — unified LoanWhiz app   (umbrella #TBD)

A single Gradio Blocks application that supersedes the three standalone clients, presenting the LoanWhiz deal-model story as an ordered-but-navigable multi-tab experience with a persistent chat, running fully live except for the pre-warmed prospectus extraction. Includes a fix to the `ReportVerifier` Vertex-client bug and the deferred covenant time-series charts.

- **Fix ReportVerifier Vertex client init** — Change `ReportVerifier` to construct its `genai.Client` for Vertex AI (project/location) like the other primitives, so it works under ADC without `GEMINI_API_KEY` or env shims. Sequencing: parallel. Paths: `src/loanwhiz/primitives/report_verifier.py`, `tests/test_report_verifier.py`.
- **App shell + omnipresent chat + shared deal state** — Multi-tab Gradio Blocks scaffold with LoanWhiz branding, a chat panel rendered/docked on every tab, and a shared loaded-deal state object the tabs read; defines the contract the tab children build against. Sequencing: parallel. Paths: `clients/demo/app.py`, `clients/demo/shell.py`.
- **Deal Overview tab** — Cache-first deal-model display: load Green Lion, read cached `deal_model.json`, show tranches, waterfall step count, trigger names, completeness score; surface a clear "extraction pre-warmed" affordance. Sequencing: sequential. After #<shell>. Paths: `clients/demo/tabs/overview.py`.
- **Pool & Performance tab** — 3-period tape analytics with EPC/geography/arrears, absorbing the dashboard standalone's content. Sequencing: sequential. After #<shell>. Paths: `clients/demo/tabs/pool.py`.
- **Waterfall tab** — Live `CollectionsAggregator` → `WaterfallRunner`; visualise the 11-step revenue cascade and per-tranche distributions. Sequencing: sequential. After #<shell>. Paths: `clients/demo/tabs/waterfall.py`.
- **Compliance & Covenants tab (with time-series charts)** — Live `ReportVerifier` (computed vs reported) + `CovenantMonitor`, plus time-series charts overlaying 3 actual points and `CashflowProjector` base/stress paths against each trigger line. Sequencing: sequential. After #<shell>, after #<waterfall>, after #<fix-reportverifier>. Paths: `clients/demo/tabs/compliance.py`.
- **Cashflow Projection tab** — Base/stress 12-month scenarios via `CashflowProjector`: projected tranche balances, WAL per scenario. Sequencing: sequential. After #<shell>, after #<waterfall>. Paths: `clients/demo/tabs/projection.py`.
- **Chat wiring** — Connect the docked chat to the planner agent (#20) with inline citations, carrying the loaded-deal context. Sequencing: sequential. After #<shell>. Paths: `clients/demo/tabs/chat.py`.
- **Retire standalone clients + demo-day runbook** — Delete `clients/{chat,dashboard,compliance}/` standalones, update README to point at the unified app, and write the demo-day runbook (pre-warm extraction cache, launch, the click-by-click narrative, fallbacks). Sequencing: sequential. After all tab children. Paths: `clients/chat/**`, `clients/dashboard/**`, `clients/compliance/**`, `clients/demo/RUNBOOK.md`, `README.md`.
